# Enforce explicit secret management by removing deprecated secret keys and autogeneration in Camunda 8.9

- Status: accepted
- Date: 2026-02-19
- Decision-makers: Balázs

## Context and Problem Statement

The Camunda Helm chart historically auto-generated secrets (random passwords) when users did not explicitly provide them, and supported legacy secret key paths (`global.secrets.*`) alongside newer conventions. This created ambiguity about the source of truth for credentials, made upgrades fragile (auto-generated values could change across installs), and added significant maintenance burden to the chart's template logic and test surface. With 8.9 being a new major chart version where breaking changes are permitted, this was the enforcement point after a deprecation period in 8.8.

## Decision Drivers

- **Security posture**: Operators must explicitly own credential lifecycle, improving auditability and eliminating "magic" secret creation that obscured who controlled access
- **Maintainability**: Autogeneration helpers, normalization logic, and dual-path key support added substantial template complexity that compounded with each new component
- **Upgrade reliability**: Auto-generated secrets changing across installs caused silent breakage during upgrades; explicit secrets make state deterministic
- **Major version boundary**: 8.9 is the correct place to enforce breaking changes after the 8.8 deprecation-with-warning period

## Considered Options

- **Keep autogeneration behind a feature flag** — Rejected because it preserves the complexity and dual code paths that motivated the removal; users who opt in still face the same upgrade fragility
- **Extend deprecation period to 8.10** — Rejected because the deprecation warnings have been in place since 8.8, providing sufficient migration runway; delaying only accumulates technical debt
- **Replace with a built-in secret operator integration** — Out of scope for chart-level changes; users can adopt external secret stores independently

## Decision Outcome

All auto-generation logic, legacy secret key paths, and associated normalization helpers were removed from the 8.9 chart. The chart now requires users to supply secrets explicitly (via values or external secret references), establishing a single, deterministic path for credential configuration across all components.

### Positive Consequences

- Reduced template complexity: removal of autogeneration helpers and normalization code simplifies the chart and makes secret flow easier to reason about
- Deterministic upgrades: secrets no longer change unexpectedly between installs, eliminating a class of silent upgrade failures
- Clearer security boundary: operators are forced to integrate with their organization's credential management practices, improving audit trails

### Negative Consequences

- Migration burden: existing users relying on autogenerated secrets must explicitly provision and supply credentials before upgrading to 8.9
- Reduced out-of-box convenience: new installations require more upfront configuration, increasing time-to-first-deploy for evaluation scenarios
