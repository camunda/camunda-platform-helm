## Summary

Following on from #6039 + #6040 introducing \`global.tls.caBundle.secret.*\`, this PR formally deprecates the legacy per-component JKS truststore fields. The legacy path stays functional (precedence is "legacy JKS wins when set" so existing deployments don't regress on upgrade), but emits a deprecation warning at install time.

## Why deprecate

- **caBundle reaches every Java component from one global field** vs. one \`tls.secret\` block per component (orchestration secondaryStorage + optimize database, ES + OS = up to 4 places).
- **PEM is text and passwordless**; JKS is binary and requires keytool + a password.
- **The JVM still consumes a keystore at runtime**, but the chart now builds it from the PEM at pod start (init container, see [helm#3498](https://github.com/camunda/camunda-platform-helm/issues/3498)). Customers never touch keytool.

## Fields marked deprecated

All still functional, all emit the warning when set:

- \`global.elasticsearch.tls.secret.{existingSecret,existingSecretKey}\`
- \`global.elasticsearch.tls.jks.secret.*\` (chart 14.x; password-injection block from [#5994](https://github.com/camunda/camunda-platform-helm/pull/5994))
- \`global.opensearch.tls.secret.{existingSecret,existingSecretKey}\`
- \`global.opensearch.tls.jks.secret.*\`
- \`orchestration.data.secondaryStorage.elasticsearch.tls.secret.*\`
- \`orchestration.data.secondaryStorage.opensearch.tls.secret.*\`
- \`optimize.database.elasticsearch.tls.secret.*\`
- \`optimize.database.opensearch.tls.secret.*\`

## Implementation

1. **Constraint helper** at \`templates/common/constraints.tpl\` extended to cover the four secondaryStorage/database paths and the two \`jks.secret\` blocks (the file already deprecated the two \`global.*.tls.secret\` paths). Updated the migration message to point at \`global.tls.caBundle.secret.*\` with a doc link to the migration recipe in [docs/self-managed/deployment/helm/configure/tls.md](https://docs.camunda.io/docs/self-managed/deployment/helm/configure/tls#legacy-per-component-jks-truststore-deprecated).
2. **\`values.yaml\` annotations** — fields tagged \`(DEPRECATED in chart 15.x — use global.tls.caBundle.secret …)\` following the chart's existing @-annotation convention, so the deprecation surfaces in the autogenerated values reference and \`helm-values-mcp\` lookups.
3. **Sanity tests** — chart renders cleanly when each legacy field is set; chart renders cleanly when \`caBundle\` and a legacy field are set simultaneously.

Note on test coverage: warning-content assertion is not possible via the testhelpers framework — \`helm template\` does not surface NOTES.txt content, which is where the constraint-warning helper emits. Same constraint applies to the existing \`global.elasticsearch.tls.secret\` and Bitnami subchart deprecation warnings, which use the same pattern. Confidence on the warning text comes from manual \`helm install --dry-run\` verification, which I ran end-to-end before this PR.
